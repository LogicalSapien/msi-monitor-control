# v0.2.3 — PBP Edge-Switch KVM — Design

**Date:** 2026-06-22
**Status:** Design — awaiting human approval. No app code until approved.
**Owner:** `msi-mac` (macOS drives design; Windows follows same model).
**Related:** `docs/PROTOCOL.md` (KVM bytes), `docs/SETTINGS.md` (config contract).

---

## Goal

When the monitor is in PBP (side-by-side) mode, automatically switch the KVM to
follow the cursor as it crosses the centre divider between the two windows — so
keyboard and mouse follow the user's gaze, Synergy-style, without manually pressing
a KVM hotkey.

**This is opt-in. OFF by default. Active only while PBP mode is on.**

## 1. Global Mouse Tracking

### 1.1 macOS — `CGEventTap`

**Chosen approach: `CGEventTap` with `kCGEventMouseMoved`.**

```
CGEventTap(
    tap:     kCGSessionEventTap,   // see all sessions
    place:   kCGHeadInsertEventTap,
    options: kCGEventTapOptionListenOnly,  // passive — no event modification
    eventsOfInterest: CGEventMask(1 << kCGEventMouseMoved),
    callback: ...,
    userInfo: selfPtr
)
```

- **`kCGEventTapOptionListenOnly`** — the tap only observes; it never consumes or
  modifies events. This is important for security and trust: we never intercept
  clicks or keystrokes.
- The tap fires on every mouse-moved event. To limit CPU, the callback does
  **nothing** unless the feature is enabled AND the monitor is in PBP mode (the
  active-state machine check, §3). When active, it reads the cursor x-coordinate
  and compares to the divider threshold — a trivially cheap operation.
- **Polling alternative (`NSEvent.addGlobalMonitorForEvents`):** the NSEvent global
  monitor is simpler (no CFRunLoop manipulation) but fires on the main thread and
  arrives slightly later. `CGEventTap` runs on a dedicated thread at a lower level,
  is more reliable for rapid cursor movement, and is the standard approach for
  pointer-location tools. It is the chosen approach.
- **CPU cost:** `CGEventTap` at `kCGEventMouseMoved` fires frequently during cursor
  movement, but the callback body is a branch check + a compare — nanosecond cost
  per event. When the feature is disabled or PBP is off, the callback returns
  immediately (`return Unmanaged.passRetained(event)`). No measurable CPU footprint
  in the idle or single-window case.
- **Thread:** the tap callback runs on a dedicated `CFRunLoop` (a background thread
  we own). The divider comparison is lock-free (the threshold is a simple stored
  value). KVM sends are dispatched to the main thread (`DispatchQueue.main.async`)
  to stay consistent with the rest of `DeviceState`.

#### macOS Accessibility / Input Monitoring permission

`CGEventTap` at `kCGSessionEventTap` requires the **Input Monitoring** permission
(System Settings → Privacy & Security → Input Monitoring, macOS 10.15+). Without
it, `CGEventTapCreate` returns `nil` and the feature is silently unavailable — the
OS does not prompt automatically.

The app must:

1. **Detect** on startup (and when the user enables the toggle) whether the process
   has Input Monitoring permission: call `CGEventTapCreate` with a throwaway
   mask; if it returns `nil`, permission is denied.
2. **Not prompt on launch.** Permission is only needed when the user explicitly
   enables edge-switch. Do NOT request it until the toggle is turned on.
3. **When the user enables the toggle:** check permission first.
   - If granted → install the tap → begin tracking.
   - If denied → show an in-settings advisory (see §5) and open System Settings to
     the Input Monitoring pane:
     `NSWorkspace.shared.open(URL(string: "x-apple.systempreferences:com.apple.preference.security?Privacy_ListenEvent")!)`
     Do NOT enable the toggle — leave it off until the user grants permission and
     re-clicks the toggle.
4. **Graceful revocation:** if the user revokes permission while the tap is live,
   `CGEventTap` fires the callback with an `enable` parameter of `false`. The tap is
   then invalidated; the app detects this and disables the feature (sets the tracking
   state to idle, updates the toggle state in the UI).
5. **Sandbox note:** this app is NOT sandboxed (unsigned desktop utility). If a
   sandboxed distribution is ever considered, `CGEventTap` requires the
   `com.apple.security.temporary-exception.mach-lookup.global-name` entitlement —
   flag this as a future concern only.

**Risk: this is the highest-risk item.** Users who decline the permission prompt
cannot use this feature at all. The Settings UI must make this clear upfront (§5).
We do not fall back to polling `NSEvent.mouseLocation` because that is macOS-version-
dependent and can be unreliable across Spaces; and it does not trigger an OS-level
permission prompt, so its behaviour on high-security machines is inconsistent.

### 1.2 Windows — Low-Level Mouse Hook (`WH_MOUSE_LL`)

**Chosen approach: `SetWindowsHookEx(WH_MOUSE_LL, ...)` with `WM_MOUSEMOVE`.**

```csharp
_mouseHook = SetWindowsHookEx(WH_MOUSE_LL, MouseProc, IntPtr.Zero, 0);
```

- The low-level mouse hook fires on `WM_MOUSEMOVE` system-wide, on the thread that
  installed it. Windows provides the cursor position in screen coordinates
  (`MSLLHOOKSTRUCT.pt`) — no additional API call needed.
- **Alternative — `GetCursorPos` polling timer:** polling every 50–100 ms in a
  `System.Windows.Forms.Timer` would be simpler and avoids the hook, but introduces
  up to 100 ms latency at the divider, which is perceptible. The hook is the correct
  approach for real-time detection.
- **CPU cost:** same as macOS — the hook fires on every mouse-move but the handler
  body is a branch check + compare. When inactive, returns `CallNextHookEx`
  immediately. No measurable cost.
- **Thread:** `WH_MOUSE_LL` requires a **message-pumping thread**. The existing
  Windows Forms message loop in `TrayApp` (which already owns the WinForms thread)
  is the right place to install the hook — same thread that owns the hotkey
  registration. The KVM send is dispatched via `Control.BeginInvoke` or
  `Application.OpenForms[0].BeginInvoke` (same idiom as any inter-thread call in the
  existing tray app), which dispatches back to the UI thread where `_device.Send`
  runs.
- **Permission:** `WH_MOUSE_LL` requires no special permission on Windows. It works
  in any normal user-mode process. No UAC elevation needed.

## 2. Identifying the MSI Display + Divider X-Coordinate

Both platforms must answer: *which screen is the MD342CQP, and where is its centre x?*

The MD342CQP is a **34-inch ultrawide (3440 × 1440)**. In PBP mode, the two windows
each occupy half the panel width — the divider is at x = screen\_origin\_x +
(screen\_width / 2).

### 2.1 macOS — `CGDirectDisplayID` matching

```swift
// Enumerate all displays; match by screen dimensions (3440×1440).
for screen in NSScreen.screens {
    let frame = screen.frame          // points (logical); multiply by backingScaleFactor for pixels
    if Int(frame.width * screen.backingScaleFactor) == 3440 &&
       Int(frame.height * screen.backingScaleFactor) == 1440 {
        msiScreen = screen
        break
    }
}
```

- **Divider x:** `msiScreen.frame.origin.x + msiScreen.frame.width / 2` (in points;
  `CGEvent` position is also in points at the session level).
- **Multi-monitor:** the cursor position from `CGEventTap` is in global screen
  coordinates. The app checks whether the cursor's x falls within the MSI display's
  frame before comparing to the divider — if the cursor is on a different monitor,
  no action is taken.
- **Resolution assumption caveat:** matching by exact pixel dimensions is fragile if
  the user runs the monitor at a non-native resolution or has another 3440×1440
  display. `CGDisplayModelNumber` / `CGDisplayVendorNumber` (IOKit or
  `CGDisplayCreateUUIDFromDisplayID`) can provide a hardware fingerprint — document
  this as a known limitation and an improvement path. For v0.2.3, match by
  3440×1440 first; if no match, the feature gracefully stays inactive (logs a
  diagnostic to the debug log).
- **Display disconnect / resolution change:** `NSScreen.screens` is re-read on each
  wakeup (the tracker caches the found screen ref; a `NSApplicationDidChangeScreenParametersNotification`
  observer re-runs the scan).

### 2.2 Windows — `Screen.AllScreens` matching

```csharp
// Match by 3440×1440 bounds.
foreach (var screen in Screen.AllScreens)
{
    if (screen.Bounds.Width == 3440 && screen.Bounds.Height == 1440)
    {
        _msiScreen = screen;
        break;
    }
}
```

- **Divider x:** `_msiScreen.Bounds.Left + _msiScreen.Bounds.Width / 2`.
- Cursor from `WH_MOUSE_LL` is in virtual screen coordinates (same space as
  `Screen.Bounds`). Same bounds check as macOS before divider comparison.
- **DPI:** `WH_MOUSE_LL` reports physical pixels. `Screen.Bounds` on a DPI-aware
  process also reports physical pixels when the app is per-monitor DPI aware
  (declared in the manifest). The app is already per-monitor DPI aware (WinForms
  default in .NET 8) — no scaling needed.
- Same `SystemEvents.DisplaySettingsChanged` handler to re-scan on resolution or
  topology change.

## 3. Enable / Active State Machine

The tracker has **two independent switches**:

| Setting toggle | PBP mode (last-sent) | Tracker state |
|:---------------|:---------------------|:--------------|
| Off            | any                  | **Idle** — hook/tap not installed |
| On             | Off or PIP           | **Standby** — hook/tap installed, callback exits immediately |
| On             | PBP (`pbpOn`)        | **Active** — divider comparison runs, KVM switches fire |

### 3.1 Transitions

- **User enables toggle (setting → On):**
  - macOS: request/check Input Monitoring permission (§1.1). If granted, install the
    `CGEventTap` → enter Standby or Active depending on current PBP mode.
  - Windows: install `WH_MOUSE_LL` → Standby or Active.
- **User disables toggle (setting → Off):**
  - Remove the hook/tap. → Idle.
- **`pbpOn` command sent successfully:**
  - If tracker is in Standby → enter Active.
  - If tracker is Idle → no change.
- **`pbpOff` or `pbpPIP` command sent successfully:**
  - If tracker is in Active → enter Standby.
- **macOS permission revoked while in Standby/Active:**
  - The tap invalidation callback fires → enter Idle + disable the toggle in the UI.

### 3.2 How the app knows PBP mode

The app tracks PBP mode via `DeviceState.currentByGroup[.pbpMode]` (macOS) /
`TrayApp._lastPbpMode` (Windows) — the in-memory last-sent tracker already present
in v0.2.2. The edge-switch tracker reads this value to decide whether to be Active
or Standby.

**OSD-staleness caveat (important — document in UI):** if the user switches PBP mode
via the monitor's physical OSD buttons (not through the app), the app's in-memory
tracker is stale. The edge-switch feature will therefore think PBP mode is whatever
the app last sent, which may be wrong. For example: the user turns PBP ON via OSD,
but the app thinks it's off → the edge-switch stays in Standby, doing nothing. This
is a known limitation of the no-state-readback constraint (PROTOCOL.md §no
read-back). Document it clearly: "Edge-switch only activates if PBP mode was last
set via this app. Changes made via the monitor's OSD are not detected."

## 4. Input → KVM Mapping

When the cursor crosses the divider, the app determines which KVM port to activate
based on which input is shown in the target window. The PBP source config (two
`InputEnum` values for main/sub windows) is already tracked in the Settings UI
dropdowns (v0.2.2).

The mapping logic:

```
inputEnum → kvmSource:
  .typeC      → kvmUSBC        (USB-C input → USB-C KVM port — direct match)
  .displayPort → kvmUpstream   (DP is typically from the upstream/PC → Upstream KVM)
  .hdmi1      → nil (no switch — see §4.1)
  .hdmi2      → nil (no switch — see §4.1)
```

### 4.1 HDMI/DP mapping ambiguity

The KVM has three ports: Auto, Upstream, USB-C. The monitor's KVM routes USB
peripherals to the host connected on whichever port is selected.

- **Type-C** is unambiguous: the USB-C cable carries both video and USB data. If a
  window shows Type-C input, the KVM USB-C port connects to that machine. → `kvmUSBC`.
- **DisplayPort** is typically the machine connected via the DisplayPort cable to the
  Upstream USB-B port on the monitor. MSI's KVM design pairs DP/HDMI with the
  Upstream USB port. → `kvmUpstream` is the best-effort mapping.
- **HDMI 1 / HDMI 2** — the monitor has only two KVM ports (USB-C + Upstream). If a
  window shows HDMI input, it is ambiguous: the HDMI source could be on either USB
  port (or neither — a game console, TV box, etc. would have no KVM relevance).
  **Decision: no automatic KVM switch for HDMI inputs.** The cursor crossing to an
  HDMI window fires no KVM command. Document this in the UI: "Auto-switch only
  applies when the target window shows a Type-C or DisplayPort source."

### 4.2 Mapping table (definitive)

| Window input | KVM switch | Notes |
|:-------------|:-----------|:------|
| Type-C       | → `kvmUSBC` | Direct hardware pairing |
| DisplayPort  | → `kvmUpstream` | Best-effort; assumes DP host is on Upstream USB |
| HDMI 1       | no switch  | Ambiguous; user must switch KVM manually |
| HDMI 2       | no switch  | Ambiguous; user must switch KVM manually |

When no switch applies (both windows HDMI, or cursor on HDMI side), the tracker
silently does nothing. The UI explains this constraint.

### 4.3 Source config access

The tracker reads the same source configuration shown in the Settings dropdowns.
These are `@State` UI-only in v0.2.2 — in v0.2.3 they must be **promoted to
`DeviceState` / a shared store** so the tracker can read them without touching UI.

Concretely:
- macOS: add `var pbpMainSource: InputEnum` and `var pbpSubSource: InputEnum` to
  `DeviceState` (or to a new `PBPState` struct within it). The Settings UI writes to
  these; the edge-switch tracker reads them.
- Windows: add `_pbpMainSource: CommandKind` and `_pbpSubSource: CommandKind` fields
  to `TrayApp`. The settings form writes them; the hook handler reads them.

## 5. Hysteresis / Debounce

A naive threshold at x = divider fires rapidly when the cursor hovers near the
boundary. Two techniques combined:

### 5.1 Dead zone (spatial hysteresis)

Define a **dead zone** of ±N pixels around the divider. The tracker maintains a side
state (`currentSide: .left | .right`). A switch fires only when the cursor moves
from one side to the other AND exits the dead zone on the new side:

```
if cursor.x < (divider - deadZone):   side = .left
if cursor.x > (divider + deadZone):   side = .right
// if cursor.x in [divider-deadZone, divider+deadZone]: no change to side
```

A KVM switch fires only on a **change** in `currentSide`. If the cursor enters the
dead zone and then retreats back to its original side without ever exiting the dead
zone on the far side, no switch fires.

**Recommended dead zone:** 48 pixels (≈ 1.4% of the 3440 px panel width, or roughly
one finger-width of mouse movement). This is wide enough to suppress jitter but
narrow enough to feel responsive. Make it a constant in the implementation — do NOT
expose it as a user-settable value in v0.2.3 (YAGNI).

### 5.2 Minimum dwell time (temporal hysteresis)

After a KVM switch fires, suppress further switches for **800 ms**. This prevents
double-firing if the user makes a quick back-and-forth gesture:

```
var lastSwitchTime: CFAbsoluteTime = 0
// in callback:
let now = CFAbsoluteTimeGetCurrent()
if now - lastSwitchTime < 0.8 { return }
lastSwitchTime = now
// fire KVM switch
```

**Why 800 ms:** a KVM switch takes the monitor a fraction of a second to complete;
re-triggering it before it settles is wasteful and confusing. 800 ms is long enough
to be imperceptible in normal use (the user's cursor has well and truly crossed) but
short enough to recover quickly from a real second crossing.

### 5.3 Combining the two

The dead zone suppresses switches during slow, tentative movement near the boundary.
The dwell time suppresses rapid back-and-forth. Together they produce robust
hysteresis without false positives.

## 8. Privacy Note

Global mouse tracking is a sensitive capability. The app must be transparent:

- **Opt-in, off by default.** The user explicitly enables it.
- **Local only.** Cursor position data is read within the process and never
  transmitted, stored, or shared. No network calls. No disk writes (other than the
  `edgeSwitchEnabled` boolean in `settings.json`).
- **Minimal scope.** Only the cursor x-coordinate relative to the divider threshold
  is used. No click data, no keystroke data.
- **macOS Input Monitoring permission** is the OS's own attestation mechanism for
  this guarantee — explain it as a feature, not an obstacle.
- The UI section includes a one-line privacy statement (§7 above). The README should
  also mention it in the features list.

## 10. Risks and Unknowns

### 10.1 macOS Input Monitoring permission (HIGH risk)

If the user declines, the entire feature is unavailable. There is no fallback
(polling `NSEvent.mouseLocation` would work but does NOT need this permission —
however it's less reliable across Spaces and Mission Control, and bypassing the
permission model is not the right approach for a public MIT-licensed app). The
correct response is clear UI: explain why the permission is needed, deep-link to the
pane, and accept that some users will not enable it.

**Mitigation:** do not ask for the permission on launch. Only ask when the user turns
the toggle on. Most users who want this feature will grant it.

### 10.2 PBP display identification (MEDIUM risk)

Matching the MSI screen by 3440×1440 resolution is fragile: it fails if the user has
two ultrawide displays, or runs the MD342CQP at non-native resolution. In those cases
the tracker logs a diagnostic and stays inactive — safe degradation, no crash. A
future improvement is to match by `CGDisplayVendorNumber` / `CGDisplayModelNumber`
(IOKit) or the Windows `EDID` data. Out of scope for v0.2.3.

### 10.3 Input→KVM mapping for HDMI sources (LOW–MEDIUM risk)

HDMI inputs have no guaranteed KVM pairing. The design explicitly does NOT switch for
HDMI (§4.2). This is correct but may surprise users who have a machine on HDMI and
expect the feature to work. UI copy must be clear: "Auto-switch applies to Type-C and
DisplayPort sources only."

### 10.4 CGEventTap reliability (LOW risk)

`CGEventTap` can be disabled by the OS if it stalls too long in the callback. The
callback body must be as lightweight as possible (branch + compare only; no
allocation, no I/O). The KVM HID send is always dispatched to the main thread
asynchronously — never done inline in the tap callback. This is already the correct
pattern.

### 10.5 Stale PBP state (LOW risk — documented caveat)

Covered in §3.2. The app cannot read monitor state, so OSD-triggered PBP changes
make the tracker out of sync. This is acceptable and must be documented (already
applies to the live-status feature in v0.2.2).

### 10.6 WH_MOUSE_LL thread requirement (Windows) (LOW risk)

`WH_MOUSE_LL` must be installed and processed on the same thread. The existing
WinForms message loop (main UI thread) satisfies this — the hook is installed in the
`TrayApp` constructor on that thread, and the message pump already runs there. No new
thread is needed.

### 10.7 Multiple KVM sends on slow cursor sweep (LOW risk)

A user dragging slowly back and forth across the divider could trigger multiple KVM
sends. The 800 ms dwell suppression (§5.2) limits this to one send per 800 ms. KVM
sends are fast and idempotent (sending `kvmUSBC` when already on USB-C is harmless).
The real cost is the HID USB round-trip — negligible.
