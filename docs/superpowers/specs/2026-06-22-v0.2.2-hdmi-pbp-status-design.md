**Date:** 2026-06-22
**Status:** Design ‚Äî awaiting human approval. No app code until approved.
**Bytes contract:** `docs/PROTOCOL.md` (updated this round) is the single source of

## Goal

Expose the now-fully-probed monitor features on both platforms:
1. **HDMI 1 / HDMI 2** main-input switching (un-park the last two inputs).
2. **PBP/PIP mode** (Off / PIP / PBP) replacing the old nil-payload PBP stubs.
3. **PBP per-window source-select** (sub-window confirmed; main-window assumed).
4. **Live "current" highlight** via last-sent tracking (monitor can't report state).

PROTOCOL.md except where noted). actionIds are the stable config-contract keys.

| actionId          | Feature      | byte[10] | Default hotkey | Notes |
|:------------------|:-------------|:---------|:---------------|:------|
| `inputHDMI1`      | `0x35 0x30`  | `0x30`   | `H`            | new ‚Äî un-parked |
| `inputHDMI2`      | `0x35 0x30`  | `0x31`   | `J`            | new ‚Äî un-parked |
| `inputDP`         | `0x35 0x30`  | `0x32`   | `D`            | existing |

### 1.2 pbpOn/pbpOff mapping decision (RECOMMENDED)

Keep the existing actionIds `pbpOn` / `pbpOff` (don't rename ‚Äî that would orphan
saved bindings) and **give them real payloads**: `pbpOff` = mode Off (`0x36 0x30`,
`0x30`), `pbpOn` = mode PBP (`0x36 0x30`, `0x32`). Add **`pbpPIP`** = mode PIP
(`0x31`) as a new actionId. This preserves the v0.2.1 P/O hotkeys (they become live)
and adds I for PIP. The optional 2nd-PBP-variant (`0x33`) is **not** exposed in
v0.2.2 (YAGNI; can add later).

### 1.3 PBP source-select ‚Äî NOT plain Command cases

Source-select is parameterised (window √ó input enum), not a fixed action, so it does
NOT fit the single-value `Command` enum cleanly. Proposal: a small separate API on
the device layer, e.g. `setPBPSource(window: .main | .sub, input: InputEnum)` that
builds the `0x36 0x31` / `0x36 0x32` payload. It is driven from the Settings UI
dropdowns (¬ß3), not from a hotkey or the main menu. `0x36 0x32` (main) is flagged
**unverified** in the UI and code.

## 4. Live "current selection" status (last-sent tracking)

The monitor does **not** report state (`IOHIDDeviceGetReport` ‚Üí `kIOReturnUnsupported
0xE0005000`), so we track the **last successfully-sent** command per *group* and show
a checkmark/highlight:
- Track three independent "current" values: **current input** (HDMI1/2/DP/TypeC),
  **current KVM** (Auto/Upstream/USB-C), **current PBP mode** (Off/PIP/PBP).
- Updated only on a `.success` send (not on failure). Persisted? **No** ‚Äî keep it
  in-memory (session only); a stale highlight across launches would be misleading.
- The menu + settings show the tracked current with a ‚úì / filled state.
- **Caveat shown to the user:** a tooltip/footnote ‚Äî "reflects the last switch made
  by this app; may be out of date if you used the monitor's OSD buttons." (The
  monitor genuinely can't tell us, so this is the honest best-effort.)

Implementation: a small `@MainActor` observable (e.g. extend `DeviceState` or a new
`StatusTracker`) holding the three current-values; `DeviceState.send` updates the
relevant one on success. UI reads it.
